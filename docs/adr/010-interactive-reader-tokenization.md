# 010. Интерактивный ридер: токенизация и стратегия перевода

**Статус:** Принято

**Дата:** 2026-04-05

## Контекст

Ридер текстов должен позволять пользователю кликать на отдельные слова и выделять произвольные фрагменты (слово, фраза, предложение), чтобы получить перевод. Текст хранится в БД как сплошная строка; для интерактивности каждый токен должен стать отдельным DOM-элементом с обработчиком событий.

Ключевые требования:
- Клик или выделение одного слова → перевод этого слова
- Выделение нескольких слов мышью → перевод выделенного фрагмента
- Пунктуация и пробелы должны отображаться корректно, не теряться
- Минимальное количество запросов к внешнему API переводов

## Рассмотренные варианты (токенизация)

- **Вариант A: токенизация на сервере** — API возвращает не текст, а массив токенов с метаданными. Плюсы: переиспользуемость. Минусы: усложняет модель данных, медленнее для больших текстов, избыточно при простой разбивке.

- **Вариант B: токенизация на клиенте (React-компонент)** — текст разбивается на токены прямо в браузере через регулярное выражение. Плюсы: не требует изменений в API, мгновенно, легко тестировать в изоляции. Минусы: логика разбивки живёт на клиенте, но для английского языка она проста и стабильна.

## Решение

Выбран **Вариант B**: токенизация на клиенте в React-компоненте `TextReader`.

### Токенизация

Текст разбивается регулярным выражением на токены двух типов:
- `word` — последовательность `\w+` (буквы и цифры)
- `separator` — пробелы, знаки препинания

Каждый `word`-токен рендерится как `<span>` с обработчиком мыши. `separator`-токены — как обычный текст без интерактивности.

### Стратегия получения перевода (два режима)

**Одно слово** (клик или выделение единственного токена):
1. Ищем слово в локальном словаре пользователя (загружается при открытии текста)
2. Если слово есть в словаре — показываем сохранённый перевод мгновенно, без запроса к API
3. Если слова нет в словаре — делаем запрос к `POST /translation`

**Несколько слов** (выделение диапазона токенов):
1. Всегда делаем запрос к `POST /translation`
2. Фразы целиком не хранятся в словаре → локальный кэш не применяется

Это решение экономит лимит внешнего API: большинство слов из текста пользователь постепенно добавляет в словарь, и повторные просмотры работают полностью офлайн.

## Последствия

### Положительные
- API остаётся простым: `GET /texts/:id` возвращает обычную строку
- Перевод одного известного слова — мгновенный, без сетевого запроса
- Слова можно визуально подсвечивать по уровню знания (данные из словаря уже загружены)
- Логика ридера полностью тестируется без бэкенда

### Отрицательные / компромиссы
- Разбивка по `\w+` не распознаёт составные слова через дефис как единое слово (например, `well-known` → два токена). Для MVP приемлемо.
- Словарь пользователя загружается целиком при открытии текста — при большом словаре (тысячи слов) может потребоваться ленивая загрузка или индексирование на клиенте.

### Риски
- Лимиты MyMemory API (500 запросов/день без ключа) могут исчерпаться при активной работе с многословными выделениями. Решение: добавить дебаунс и persist-кэш в localStorage в будущем.
